Content style guide

This guide covers style, spelling and grammar conventions for all content published by Department for Work and Pensions (DWP). These patterns are based on the research and experience of the DWP content community.

You should use this guide to build on the GOV.UK style guide and DWP design patterns. If you can’t find the pattern you’re looking for here, we’re discussing more on the DWP content channel in Slack.

c

Citizens - mainstream users of services, using 'people' is better.

Common case - design for the common case, this means everyone who's at a certain point, not the majority. If there's a significant minority at a certain point, they need something or should have been branched off before getting here.

d

DWP - don't use the definite article when writing out in full, it's simply Department for Work... not THE Department for Work... Avoid using 'we' to refer to DWP, be more active.

Digital by default - use 'if you can't get online' to promote digital by default, rather than preference for other channels.

i

If - if important and results in branching then better to start with conditionals, people reading them can then discount the rest if not relevant

j

Jargon - if unavoidable then define using scare quotes, eg 'money to help you with difficulties at work (known as an 'Access to Work' grant)'

k

l

Licence - is a noun

License - is a verb

Lists - introduce lists with a lead that means each item would form a full sentence in isolation. If the list introduces a selection and this isn't possible use, 'one of the following:'

m

May - usually can be replaced with 'can'

Might - people prefer certainty, but if you can't be certain than use might as long as you caveat it with a conditional. "You might be able to x, if you y"

Mr, Mrs and other titles - always question the need for them, try to avoid

n

o

OK - it's fine to use it

p

People - the preferred term for citizens or users.

Please - don't use please (or thank you). It reduces authority and is insincere.

Pronouns - users of services are referred to in the second person apart from in legal declarations when you should use the first person. Avoid using plural pronouns (we and us) for government unless not doing so would result in a passive construction. Preferably keep the user as the focus, eg say 'you'll get' rather than 'we'll send'.

Provide - banned word. Use give, send or something more active instead.

q

r

s

Sorry - only say sorry when something has gone wrong that's the fault of the service. For example a 500 page error because of server outage.

Statement answers - this type of selection answer in interactions often works better than yes and no, particularly if there's more than one clause in the question or more than 2 common responses in certain circumstances, for example 'Do you have a bank account?', 'Yes you have a bank account', 'No, but you plan to open one', 'No, you are unable to open one'.

t

Thank you - use sparingly, only on completion pages.

Timescales - people seem to prefer being given timescales to specifics, eg 'in 4 working days' worked better than 'by next Tuesday' on Access to Work. Don't put timescales in for the sake of it, 'you'll get a call in the next 24 hours' is unlikely to have much effect as users aren't that impatient.

u

Users - refer to people as users when you're talking about research observations that are specific to a particular service, eg 'users of Carer's Allowance prefer to put their name in separate fields', otherwise use 'people' or 'citizens'.

v

w

x

y

Yes - if using yes and no constructions be aware users usually want to answer 'yes' - design with data and make the most common case the 'yes' path unless you have user evidence it doesn't work - see statement answers
